# Тестирование

## Стек

Тесты написаны на **Qt Test** (`Qt6::Test`). Преимущества для проекта на Qt:

- `QSignalSpy` ловит Qt-сигналы без обвязки;
- `QTRY_VERIFY` / `QTRY_COMPARE_WITH_TIMEOUT` ждут асинхронные условия в цикле событий;
- `QTEST_MAIN`/`QTEST_APPLESS_MAIN` подставляют корректный `QCoreApplication` (или работают без него);
- интеграция с `ctest` "из коробки" через `add_test`.

## Запуск

Тесты собираются опционально (см. [Сборка и интеграция](08-Сборка-и-интеграция.md)):

```sh
cmake -S . -B build/Desktop-Debug -DGCHANNELMANAGER_BUILD_TESTS=ON
cmake --build build/Desktop-Debug
cd build/Desktop-Debug
LD_LIBRARY_PATH="$PWD" ctest --output-on-failure
```

Ожидаемый вывод:

```
Test project .../build/Desktop-Debug
    Start 1: SimpleFrameCodec
1/2 Test #1: SimpleFrameCodec .................   Passed    0.01 sec
    Start 2: Gateway
2/2 Test #2: Gateway ..........................   Passed    0.33 sec

100% tests passed, 0 tests failed out of 2
```

Запуск одного теста для подробного вывода:

```sh
./tests/tst_Gateway -v2
./tests/tst_Gateway -v2 -fn "session_*"   # фильтр по имени
```

## `FakeTransport`

Главный тестовый инструмент. Не делает ввода-вывода, но честно соблюдает контракт `ITransport`:

| Возможность | Метод |
|---|---|
| Запоминает всё отправленное | `[[nodiscard]] const QList<QByteArray> &sent() const` |
| Доставляет произвольные байты "от узла" | `void simulateReceive(const QByteArray &bytes)` |
| Имитирует ошибку транспорта | `void simulateError(const QString &msg)` |
| Очищает буфер отправленного | `void clearSent()` |
| Дополнительный сигнал для проверок | `signal void dataSent(const QByteArray &)` |

> [!TIP]
> Конкретный паттерн "ждать, пока появится Request-кадр" в тестах: `QTRY_VERIFY(lastRequestCorrId(t) != 0);` — `lastRequestCorrId` локальный хелпер в `tst_Gateway.cpp`, который перебирает `t->sent()` и парсит фреймы через `SimpleFrameCodec::parse`.

## Что покрыто

### `tst_SimpleFrameCodec` (15)

- `encodeRequest_roundtrip` — `Reply` кадр с тем же `corrId` парсится как `Type::Reply`.
- `incomingRequestFrame_classifiedAsTypeRequest` — `Request`-кадр классифицируется как `Type::Request` с сохранением `corrId`.
- `encodeReply_carriesCorrelation` — `Reply`-кадр несёт тот же `corrId`, что и запрос.
- `encodeData_carriesNoCorrelation` — `Data`-кадр, `corrId == 0`, тип `Type::Data`.
- `keepAlive_reqDecodesAsPing_replyDecodesAsKeepAlive` — `KeepAliveReq` → `Type::KeepAlivePing`, `KeepAliveReply` → `Type::KeepAlive`.
- `encodeSessionStart_classifiedAsType_SessionStart` — `SessionStart`-кадр → `Type::SessionStart`.
- `encodeSessionStartAck_classifiedAsType_SessionStartAck` — `SessionStartAck`-кадр → `Type::SessionStartAck`.
- `encodeSessionStop_classifiedAsType_SessionStop` — `SessionStop`-кадр → `Type::SessionStop`.
- `feed_splitAcrossCalls_buffered` — кадр, разрезанный по середине между двумя `feed()`, корректно склеивается.
- `feed_garbagePrefix_resyncsByMagic` — мусор перед `0xA5` пропускается посимвольно.
- `feed_multipleFrames_inOneBuffer` — три кадра в одном вызове `feed()` → три `DecodedMessage`.
- `parse_consumesBytesIncrementally` — `parse()` удаляет потреблённое из буфера, "хвост" остаётся.
- `reset_clearsBuffer` — `reset()` обнуляет внутренний буфер.
- `feed_oversizedLength_resyncsWithoutOverrun` — заголовок с огромным `len` отвергается (без OOB-чтения / бесконечной буферизации), парсер ресинхронизируется.
- `feed_corruptedFrame_droppedByCrc` — перевёрнутый бит в payload не проходит хвостовой CRC; кадр отбрасывается, а не доставляется как валидные данные.

## Паттерны для собственных тестов

### Готовая обвязка

```cpp
FakeTransport *wireUp(Gateway &gw,
                      std::chrono::milliseconds ka = std::chrono::milliseconds(40),
                      bool keepAliveEnabled = true);
```

В тестах удобно начинать с этого хелпера — он создаёт кодек, фейк-транспорт и применяет keep-alive конфиг.

### Ожидание Active

```cpp
auto *t = wireUp(gw);
gw.enableChannel();
gw.startSession();
// SessionStart уже улетел; пришёл SessionStartAck — сессия Active
t->simulateReceive(SimpleFrameCodec::makeFrame(SimpleFrameCodec::SessionStartAck, 0, {}));
QCOMPARE(gw.sessionState(), Gateway::SessionState::Active);
```

### Round-trip запроса

```cpp
auto *req = gw.sendRequest(QByteArray("ping"));
QSignalSpy ok(req, &GatewayRequest::succeeded);

QTRY_VERIFY(lastRequestCorrId(t) != 0);
const quint32 cid = lastRequestCorrId(t);
t->simulateReceive(SimpleFrameCodec::makeFrame(SimpleFrameCodec::Reply, cid,
                                               QByteArray("ACK")));
QTRY_COMPARE(ok.count(), 1);
```

### Ожидание Suspended через timeout

Прямого хука нет — нужно дать keep-alive таймеру отработать `maxMissed + 1` тиков без ответа. Если в тесте интервал keep-alive = 40 мс, `maxMissed = 3`, то к ~160 мс наступит `Suspended`:

```cpp
QTRY_COMPARE_WITH_TIMEOUT(gw.sessionState(),
                          Gateway::SessionState::Suspended, 1000);
```

## Стиль ассертов

Внутри `private slots:` методов используются:

| Макрос | Когда |
|---|---|
| `QCOMPARE(actual, expected)` | сравнение значений |
| `QVERIFY(cond)` | произвольное условие |
| `QTRY_VERIFY(cond)` | то же, но крутит цикл событий до 5 с (можно изменить `QTRY_VERIFY_WITH_TIMEOUT`) |
| `QTRY_COMPARE(actual, expected)` | то же для сравнений |
| `QSignalSpy` | подсчёт/анализ сигналов |

`QTRY_*` — основное оружие в тестах с таймерами. Не пишите `QThread::msleep` в тестах — он блокирует цикл событий и ломает обработку сигналов.
