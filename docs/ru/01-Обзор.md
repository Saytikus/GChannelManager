# Обзор

## Что это

`GChannelManager` решает типовую задачу: у вас есть произвольный байтовый транспорт (последовательный порт, UDP-сокет, кастомный RUDP-стек) и собственный протокол кадров. Нужно надёжно отправлять команды, получать ответы, переживать кратковременные обрывы линии и видеть метрики связи. Библиотека предоставляет:

- **слой канала** — открытие/закрытие транспорта;
- **слой сессии с keep-alive** — переход в `Suspended` при потере heartbeat и возврат в `Active` без переоткрытия канала (поведение в духе RUDP);
- **запрос/ответ с корреляцией** — `sendRequest(payload) → GatewayRequest`, с автоматическими повторами и экспоненциальным backoff;
- **fire-and-forget отправка** — `send(payload)`, без ожидания ответа;
- **статистика** — счётчики байт/запросов/heartbeat'ов с периодическим сигналом.

Библиотека **не знает** ваш протокол и ваш транспорт. Эти две зависимости вы подключаете через интерфейсы [`ITransport`](05-Транспорт.md) и [`IMessageCodec`](04-Протокол-и-кодек.md). Для иллюстрации есть `SimpleFrameCodec` — простой бинарный кадр-протокол.

## Ключевые возможности

| Возможность | Где | Подробнее |
|---|---|---|
| Включение/выключение канала | `Gateway::enableChannel()` / `disableChannel()` | [Состояния и переходы](03-Состояния-и-переходы.md) |
| Сессия с keep-alive | `Gateway::startSession()` / `stopSession()` | [Сессия](03-Состояния-и-переходы.md#сессия) |
| Keep-alive на лету | `setKeepAliveEnabled(bool)` / `setKeepAliveConfig(...)` | [Keep-alive](06-Gateway-API.md#keep-alive) |
| Устойчивость к разрывам | автоматический переход `Active ↔ Suspended` | [Состояния и переходы](03-Состояния-и-переходы.md) |
| Запрос/ответ | `sendRequest(payload)` → `GatewayRequest*` | [sendRequest](06-Gateway-API.md#sendrequest) |
| Повторы с backoff | `RetryPolicy {maxRetries, timeout, backoffFactor, maxTimeout}` | [RetryPolicy](06-Gateway-API.md#retrypolicy) |
| Fire-and-forget | `send(payload) → bool` | [send](06-Gateway-API.md#send) |
| Серверная роль | signal `requestReceived` + слот `reply(corrId, response)` | [Серверная роль](06-Gateway-API.md#серверная-роль-входящие-запросы-и-reply) |
| Кэш ответов (idempotency) | `setReplyCacheConfig(...)`, повтор запроса обслуживается из кэша | [Кэш ответов](06-Gateway-API.md#кэш-ответов-серверная-роль) |
| Статистика | `stats()`, `setStatsInterval(...)`, signal `statsUpdated` | [Статистика](07-Статистика.md) |

## Минимальный пример

```cpp
#include <GChannelManager/Gateway.h>
#include <GChannelManager/SimpleFrameCodec.h>
// ваш транспорт, реализующий ITransport:
#include "MySerialTransport.h"

Gateway gw;
gw.setCodec(std::make_unique<SimpleFrameCodec>());
gw.setTransport(std::make_unique<MySerialTransport>(SerialConfig{...}));

QObject::connect(&gw, &Gateway::sessionStateChanged,
    [&](Gateway::SessionState s) {
        if (s == Gateway::SessionState::Active) {
            auto *req = gw.sendRequest(QByteArray("HELLO"));
            QObject::connect(req, &GatewayRequest::succeeded,
                [](const QByteArray &resp) { qInfo() << "got" << resp; });
        }
    });

gw.enableChannel();   // дальше всё — событийно
```

Подробный разбор шагов — в [Руководстве пользователя](10-Руководство-пользователя.md).

## Технологии

- **C++20**, шаблоны/`concepts` не используются — обычный современный C++ с `[[nodiscard]]`, `enum class`, `std::unique_ptr`, лямбдами и `std::chrono`.
- **Qt 6 (Core)** — обязательная зависимость (моментально работает и с Qt 5 при наличии).
- **Qt 6 (Network)** — нужна только из-за `QHostAddress` в [`UdpConfig`](05-Транспорт.md#udpconfig); если UDP не нужен — соответствующий заголовок можно не подключать.
- **CMake ≥ 3.16**, сборка `Ninja`/`Make`/`MSBuild`.
- **Qt Test** — опциональный модуль (только для запуска юнит-тестов).
